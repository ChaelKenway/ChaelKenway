第二章 关系数据模型（上）

2.1 数据模型概述

数据模型是对现实世界数据特的模拟和抽象。

三要素：

  1. 数据结构（静态特性）：描述组成对象和对象间的联系。
  2. 数据操作（动态特性）：查询和更新，后者包括插入、删除和修改。
  3. 数据上的约束：约束条件是一组完整性规则（数据及其联系所具有的制约和依存规则）的集合。例如一周中每一天只能是 1-7 的整数。
    关系数据模型是重点。

其他数据模型有层次、网状。

2.2 关系模型基础

关系模型的组成

  1. 关系数据结构：数据的逻辑结构是一张二维表，由行和列组成。

    基本概念：

    1. 关系

      一个表格的名字，描述事物和事物间的联系。
      例如“关系movies”：描述电影信息的二维表。

    2. 属性

      关系的列命名。也就是第一行中每个元素。

    3. 模式

      关系名和其属性的组合。是一个无序集合。

      表示方式： 关系名（属性1， 属性2， 属性3）
      如 Movies(title, year, length, filmType)

      一项关系模型设计包括一个或多个关系模式，这个关系模式的集合叫做关系数据库模式（数据库模式）。

      如 学生（学号，姓名） 课程（课程号，课程名，学分） 选修（学号，课程名，成绩）

    4. 域

      属性的数据类型、取值范围。

      关系Movies中，title的域是字符串集合，人的年龄是0-160.

      可以把域标注在关系模式中。

      如 Movies(title: string, year: interger, length: interger, filmType: string)

    5. 元组

      关系中除去属性名（首行）外的其他行。一行是一个元组。

      元组中每一列的属性要对应第一行的每一列，且其域是相同的。

    6. 键

      是关系的某个属性（组），可以唯一确定一个元组，且若从属性组中去掉任何一个属性，他就不具有这一性质了。

      如 Movies(title, year, length, filmType) 的键是属性组(title, year)，意味着用标题和年份可以唯一确定一部电影。

      同理也可以用学号来唯一确定一个华大学生。

    7. 候选码

      如果有多个属性（组）都能唯一标识关系的元组，若从其中去掉任何一个属性，它都会失去这一性质，这些属性（组）就是关系的候选码。

      • 主属性： 包含在候选码中的属性。
      • 非主属性（非码属性）： 不包含在任何候选码中的属性。

    8. 主码（主键）

      在若干个候选码中指定的作为码（键）的属性（组）。

      每个表只能有唯一主键，且不为空。

      主键可以唯一确定一个元组，保证关系实例上任何两个元组在键上取值不同。

      标记方式： 下划线

      如 Movies(title, year, length, filmType) 的主键是属性组(title, year)。

    9. 全码

      如果一个关系的所有属性在一起才能构成关系的码，则成为全码。

    10. 外键（外部关键字）

      将一个表的主键作为数据之间联系的纽带放到另一个表中。

      例如：

      –学生表–

      学号 姓名 性别 出生年月
      01001 张三 男 1999-07
      01002 李四 男 1999-08

      “学号”是学生表的主键。

      –成绩表–

      学号 课程号 成绩
      01001 TW-MO-1 80
      01001 DM-MO-1 80
      01002 TW-MO-1 90
      01002 DM-MO-1 90

      “学号”是成绩表的外键

    11. 关系的等价描述

      关系是元组的集合，集合是无序的，所以交换列的位置后关系仍然是等价的。

    12. 关系实例

      一个给定关系中元组的集合。

      • 当前实例： 通常数据库系统仅仅维护关系的一个版本，即关系的“当前”元组集合。
      • 模式和实例的区别：
        • 模式 = 关系名（关系属性1，属性2 ……）是相对不变的
        • 实例 = 元组的集合，即取出不同的行，经常会变。

  2. 关系操作集合

    查询，插入，删除，修改等。

  3. 关系完整性约束

    • 实体完整性约束（必须）

      关系模型中主码是唯一标识，主属性不能取空。

    • 参照完整性约束（必须）

      F是关系R中非码的一个或一组属性，但又是另一个关系S的主码。则F是关系R的外码，称R为参照关系，S为被参照关系。

      R中每个元组在F上的值必须为：（参照完整性规则）

      • 空值（F的每个属性都是空）
      • 等于S中某个元组的主码值

      参照完整性规则是定义外码与主码之间的引用规则。

    • 用户定义的完整性（自行定义）

      针对某一具体的数据必须满足的语义要求。

      如 学生信息（学号，姓名，性别，年龄）中性别只能为“男”或“女”，年龄只能是大于0的整数。

2.3 在SQL中定义关系模式

SQL的特点

  • 综合统一

    SQL的核心

    • 数据定义语言

      数据库的逻辑结构，对关系模式一级的定义，包括基本表、视图和索引的定义。

    • 数据查询语言

    • 数据操纵语言

      对关系模式中的具体数据的增、删、改等操作。

    • 数据控制语言

      数据访问权限的控制。

  • 高度非过程化

    只需要知道“做什么”，无需了解存取路径

  • 用同一种语法结构提供两种使用方式

    • 自含式语言：独立用于联机交互的使用方式，用户可以在终端键盘上直接键入SQL命令对数据库进行操作。
    • 嵌入式语言：SQL语句能嵌入到高级语言程序中。
    • 但是两种方式下，SQL的语法结构基本一致。

  • 语言简洁，易学易用

    数据定义、操纵、控制的核心功能只用了9个动词：

    • CREATE
    • DROP
    • ALTER
    • SELECT
    • INSERT
    • UPDATE
    • DELETE
    • GRANT
    • REVOKE

SQL的三个主要标准

  1. ANSI（美国国家标准机构）SQL
  2. SQL-92(SQL2)
  3. SQL-99(SQL3)

基本数据类型

  1. 字符串

    • CHAR(n)

      n个字符的固定长度的字符数据，实际占用不足n个的用空格填充。最大长度为8000字符。

      索引效率高，但会浪费空间。

    • VARCHAR(n)

      最多n个字符的可变长度字符数据，存储空间是该字符数据其实长度+1。最大长度为8000字符。

    • TEXT

      可变长度字符数据，最大为(2^31-1)个字符

  2. unicode字符串

    • unicode将所有字符用两个字节表示，解决了字符集不兼容问题。缺点在于存储英文（1字节）时需要用2字节。

    • NCHAR

      固定长度，最大4000字符。

    • NVARCHAR

      可变长度，最大4000字符。

    • NTEXT

      可变长度，最大(2^30-1)字符。

  3. 二进制

    • BINARY

      固定长度，最大8000B。

    • VARBINARY

      可变长度，最大8000B。

    • IMAGE

      可变长度，可以存储照片、图片，最大(2^31-1)字节。

  4. 整型数据

    • BIGINT

      -2^63 ~ 2^63-1

    • INT

      -2^31 ~ 2^31-1

    • SMALLINT

      -2^15(-32768) ~ 2^15-1(32767)

    • TINYINT

      0 ~ 255

    • BIT（也可以模拟bool）

      1 / 0

  5. 浮点数值

    • 精确浮点型

      • DECIMAL(n, d)

        有n位数字（不含小数点）的十进制数，小数点右边有d位小数。

      • NUMERIC

        功能上等同于前者。

    • 精确浮点型

      • FLOAT

        -1.79e+308 ~ 1.79e+308

      • REAL

        -3.4e+38 ~ 3.4e+38

  6. 日期时间数据

    • DATETIME

      1753/1/1 ~ 9999/12/31的日期和时间，精确到3%秒。

      每个值要求8B.

    • SMALLDATETIME

      1900/1/1 ~ 2079/6/6的日期和时间，精确到分钟。

      每个要求4B.

  7. 货币数据

    • MONEY

      -2^63 ~ 2^63-1

      精确到货币单位的千分之十。

    • SMALLMONEY

      -214748.3648 ~ 214748.3647

      精确到货币单位的千分之十

  8. 其他数据

SQL五个基本操作

  1. 定义基本表 CREATE TABLE
  2. 修改基本表的定义 ALTER TABLE
  3. 删除基本表 DROP TABLE
  4. 创建索引 CREATE INDEX
  5. 删除索引 DROP INDEX

参数说明

  • 圆括号：内容为必选参数，可有多个选项。
  • 放口号：内容为可选项，用户根据需要使用。
  • 竖线：参数间“或”的关系。
  • 省略号：重复前面的语法单元。
  • 尖括号：要求的参数。
  • 方括号和省略号[……] 同样选项还可以重复多遍。

完整性约束条件

如果涉笔到多个属性列，则必须定义在表级上，否则可以定义在列级或表级上。

  1. NOT NULL / NULL

    前者不允许存在空值，给主属性；后者可以给非主属性。

  2. UNIQUE

    唯一性，不允许某一列或某几列有重复的属性值。

  3. DEFAULT

    默认值约束，对特定列没有指定，则使用DEFAULT后面的默认值。

  4. CHECK

    检验约束，插入值或更新至必须满足这个条件。

  5. PRIMARY KEY

    主键约束，表中的主键可以是一列或多列，此约束使得主键的数值在每一行中各不相同。约束的列不能为空。

  6. FOREIGN KEY

    外键约束，表中的一列或多列，所包含的值与同一个表或其他表中PRIMARY KEY定义的主键值相匹配。

CREATE TABLE

CREATE TABLE <表名> (

<列名><数据类型>[<列级完整性约束条件>],

<列名><数据类型>[<列级完整性约束条件>],

<列名><数据类型>[<列级完整性约束条件>],

……

[<表级完整性约束条件>] );

如果将一个字段指定为NOT NULL，则必须是有效的数据。

CREATE TABLE Movies(

​ title CHAR(100),

​ year INT,

​ length INT,

​ genre CHAR(10),

​ studioName CHAR(30),

​ producer# INT

);

ALTER TABLE

ALTER TABLE<表名>

[ADD <新列名> <数据类型> [完整性约束] ]

[DROP <COLUMN 列名> ]

[DROP <完整性约束名> ]

[ALTER COLUMN <列名> <数据类型> ];

不能同时添加或删除一个以上的字段或索引。

DROP TABLE

DROP TABLE <表名> [RESTRICT | CASCADE ];

删除<表名>指定的表。

  • 若选择RESTRICT，则要删除的表不能被其他表的约束
